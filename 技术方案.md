# LabMind — 高校课题组智能体驱动团队主页技术方案

> **项目命名**：LabMind（实验室 × 智能体）  
> **开发模式**：Vibe Coding — 你描述需求，AI 写代码，最小化手动操作  
> **部署路线**：前期 GitHub + Vercel 免费托管，后期迁移阿里云 ECS  
> **核心亮点**：基于 Harness Engineering 的多 Agent 体系，用户只需说一句话

> **2026-07-13 实现口径更新（优先于下文早期设想）**：当前所有 AI 调用已统一为 DeepSeek `deepseek-chat` + `DEEPSEEK_API_KEY`；动态类型统一为论文发表/学术动态/成员动态；Harness 已落地运行时工具白名单、输入预算、附件上限、超时/重试、去敏审计与“AI 不得直接发布”边界。成员分类为指导老师/研究成员/毕业生，管理员为可附加权限；论文和动态已有本地审核队列原型。实际目录和进度以 `AGENTS.md` 为准。

## 二、技术栈选型

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端框架** | Next.js 16（App Router） | React 生态，Vercel AI SDK 原生支持，服务端路由与前端同仓维护 |
| **UI 样式** | Tailwind CSS v4 + CSS 变量 | 深浅双主题，组件统一使用语义颜色 |
| **动画** | Framer Motion | 滚动渐入；tsParticles 已移除并删除依赖 |
| **内容管理** | Sanity Studio（免费版） | 可视化表单，成员不需要懂代码；免费额度：10k文档+100G存储 |
| **AI 框架** | Vercel AI SDK + TypeScript Harness | 当前路由直接受 `agent-harness.ts` 约束；CrewAI 仅保留为后续可选方案 |
| **大模型** | DeepSeek `deepseek-chat` | 支持 OpenAI 兼容接口；聊天、Intake 与动态生成统一接入 |
| **数据来源** | Semantic Scholar API（优先）+ DBLP + CrossRef | 均免费；SS 覆盖最广；DBLP 对 CS 权威；CrossRef 验证 DOI |
| **爬虫** | Python + httpx + APScheduler | 异步高性能；APScheduler 原生 cron 支持 |
| **审计数据库** | PostgreSQL（Supabase 免费层） | Agent 行为日志、审核队列；Supabase 前期免费 |
| **部署（前期）** | GitHub + Vercel | 零配置，每次 push 自动部署，完全免费 |
| **部署（后期）** | 阿里云 ECS + Docker Compose + Nginx | 自有服务器，数据完全可控 |

### 为什么不用 Jekyll / Hugo 纯静态？

纯静态方案需要手动编辑 YAML 文件再重新部署，**无法实现"说一句话 Agent 自动补全上传"和"定时爬取自动更新"**，故放弃。

## 四、Harness Engineering — 多 Agent 体系设计

> **核心思想**：不让 AI 自由发挥，而是给每个 Agent 套上"马具（harness）"——  
> 明确角色 + 受限工具集 + 可观测执行过程 + 可审计日志。  
> 每个 Agent 只做一件事，不可越权；AI 只产生检索结果、草稿或审核建议，最终发布必须由管理员人工确认。

## 五、三层安全防护体系

```
第一层：输入过滤（实时，Nginx 层）
  · 请求频率限制：AI 接口 10次/分/IP，普通 API 5次/秒/IP
  · 基础 XSS/注入过滤
  · 请求体大小限制（防超长 prompt 注入）

第二层：Moderation Agent 预检查（自动给建议，不直接发布）
  · Safety Check：规则检查 + DeepSeek 辅助审查，检测敏感内容
  · Academic Check：CrossRef API 验证 DOI 真实性，DBLP 交叉验证
  · Duplicate Check：哈希比对已有内容，防重复
  · Completeness Check：必填字段完整性
  · 置信度低于阈值 → 自动升级人工审核

第三层：Watchdog 持续监察（异步，独立进程）
  · 每小时扫描 Agent 操作日志，检测越权行为
  · 每天统计 Moderation 通过率/误拒率，异常时告警
  · 独立部署，与业务逻辑完全解耦
```

### 7.3 定时爬虫（Python）

```python
# crawler/scholar_sync.py
import httpx
from apscheduler.schedulers.asyncio import AsyncIOScheduler

scheduler = AsyncIOScheduler()

SEMANTIC_SCHOLAR_BASE = "https://api.semanticscholar.org/graph/v1"

async def fetch_author_papers(ss_author_id: str) -> list:
    """Semantic Scholar API 查询作者最新论文"""
    url = f"{SEMANTIC_SCHOLAR_BASE}/author/{ss_author_id}/papers"
    params = {
        "fields": "title,authors,year,venue,citationCount,doi,abstract",
        "limit": 50,
    }
    async with httpx.AsyncClient() as client:
        resp = await client.get(url, params=params, timeout=30)
        return resp.json().get("data", [])

@scheduler.scheduled_job('cron', hour=2, minute=0)      # 每天凌晨2点
async def daily_incremental_sync():
    """增量同步：只提交最近30天内的新论文"""
    members = await get_members_with_scholar_id()
    for member in members:
        papers = await fetch_author_papers(member['semanticScholarId'])
        for paper in papers:
            if is_recent(paper, days=30) and not await paper_exists(paper['doi']):
                await submit_to_review_queue(paper, source='auto_crawler')

@scheduler.scheduled_job('cron', day_of_week='sun', hour=3)  # 每周日凌晨3点
async def weekly_citation_update():
    """更新所有论文的引用数 + 成员 h-index"""
    await sync_citation_counts()
    await sync_member_hindex()
```

## 十二、风险与应对

| 风险 | 应对方案 |
|-----|---------|
| localStorage 账号/权限可被篡改 | Phase 2 第一优先级迁移服务端认证、会话、权限校验；完成前不开放写 API |
| AI 越权或伪造“已发布” | Harness 固定 `canPublish: false`；工具运行时白名单；提交与发布仅走业务页面和管理员操作 |
| 提示词注入/超长附件 | 将附件标记为不可信输入，限制格式、数量、文本长度、总输入量、步骤、重试和超时 |
| AI 补全虚假论文信息 | Moderation Agent 强制 CrossRef API 验证 DOI 真实性 |
| Google Scholar 反爬 | 优先用 Semantic Scholar（官方 API，免费无反爬）；Scholar 降级备用 |
| API 费用失控 | DeepSeek 账户设置用量告警；网关对 AI 接口限流；保留用量日志 |
| 旧版 AI SDK/Next 依赖公告 | 当前无 high/critical；Phase 2 安排主版本迁移与完整回归，不使用 `npm audit fix --force` 跨版本强改 |
| 成员误删内容 | Sanity 保留完整历史版本，一键回滚 |
| Watchdog 被攻陷 | Watchdog 零写权限，攻陷后最多丢失告警，无法篡改数据 |
| Moderation Agent 误拒 | Watchdog 统计误拒率，超阈值时通知导师人工校正策略 |

#### `lib/agent-harness.ts` — Agent 工具白名单

Harness Engineering 核心：每个 Agent 只能调用自己白名单内的工具；真实工具执行必须经过 `runGuardedTool()`，并同时受输入量、附件、步骤、重试和超时策略限制。所有 Agent 固定 `canPublish: false`。

```
AGENT_TOOL_REGISTRY = {
  orchestrator: ['classify_intent'],
  intake:       ['search_existing_*', 'fetch_doi_metadata', 'ask_user_clarification'],
  research:     ['fetch_semantic_scholar', 'fetch_dblp', 'fetch_crossref'],
  qa:           ['search_lab_info', 'guide_publication_submission', 'guide_news_submission'],
  moderation:   ['read_pending_content', 'check_content_safety', 'verify_doi_crossref',
                 'check_duplicate', 'prepare_review_recommendation'],
  watchdog:     ['read_agent_audit', 'read_published_content', 'create_admin_alert'],
}
```

## 十六、FanLearn Lab 中的借鉴与体现

> 本节梳理 FanLearn Lab 当前实现对所调研开源项目设计理念的借鉴，并标注超越现有方案的原创突破。
> **三个重点调研项目（al-folio / Lab Website Template / research-website-template）单列优先说明，其余项目影响合并表述。**

### 16.1 三个重点调研项目的借鉴（优先）

#### A. 借鉴 Lab Website Template（LWT）—— 课题组信息架构 + 论文自动查询

| 借鉴点 | 体现位置 | 说明 |
|-------|---------|------|
| **课题组成员分组** | `/team` 指导老师/研究成员/毕业生；管理员为可附加权限 | 吸收 LWT 的角色分组思路，同时将学术身份与网站管理权限分离 |
| **成员独立详情页 + 多字段** | `/team/[slug]` 左侧信息卡（邮箱、Scholar、GitHub、研究方向、加入年份） | 对应 LWT 每位成员独立详情页 + ORCID 等字段 |
| **成员-论文自动关联（aliases 模式）** | `lib/member-match.ts` + Member.aliases 字段 | 直接采用 LWT 的核心思路：作者名自动匹配，不再手动维护关联 |
| **论文自动查询** | `app/api/publications/lookup/route.ts` 调 Semantic Scholar + Crossref + DBLP；AddPublicationModal 弹窗 | 兑现 LWT "输入 DOI 或标题/作者后给出候选论文" 的核心优势；若未明确写作者，系统会把登录姓名当作弱线索，并为 Phase 3 AI 版铺路 |
| **精选论文区** | Publication.isHighlight 字段保留 | 对应 LWT 的 Highlights 区（Phase 1.5 已隐藏筛选按钮，字段保留供 Phase 2 使用） |

**关键说明**：LWT 最有价值的"一行查询补全"在 Phase 1.5/1.6 已落地——现在已从单一 DOI 扩展为标题、常见标识和多源学术检索，这是三个模板中我们直接对标并跑通的核心能力。

#### B. 借鉴 al-folio —— 论文展示体系 + 学术规范

| 借鉴点 | 体现位置 | 说明 |

| **按年份倒序分组** | `publications` 页年份筛选 + 默认按 year 倒序 | 对应 al-folio 按年份分组 |
| **论文卡展开摘要** | PaperCard 点击 ChevronDown 展开/收起 abstract | 对应 al-folio 悬停/点击展开摘要 |
| **BibTeX 批量导入** | `lib/bibtex-parser.ts` 零依赖解析器 + BibtexImportModal 弹窗 | 补齐 al-folio 的论文管理优势：粘贴 .bib 批量导入，迁移历史论文省力 |
| **左侧筛选栏** | `publications` 页 aside 侧栏（年份 + 类型） | 对应 al-folio 的筛选侧栏 |

#### C. 借鉴 research-website-template —— 现代技术栈 + 数据分离

### 16.3 FanLearn Lab 的创新突破（超越三个调研项目）

| 创新点 | 说明 | 与调研项目对比 |
|-------|------|--------------|
| **AI 对话助手（LabMind）** | `/chat` 接入 DeepSeek 流式对话，并支持文件上传上下文 | 三个调研项目均为纯静态网站，无任何 AI 能力；research-website-template 虽技术栈现代但未扩展 AI |
| **Multi-Agent Harness 架构** | `lib/agent-harness.ts` 定义 6-Agent 工具白名单，并实际约束工具执行、输入预算、附件、步数、超时、去敏审计和“AI 不得直接发布”边界 | 学术网站领域独创，无同类项目 |
| **登录 + 附加管理权限体系** | `lib/auth.tsx` AuthProvider + 游客/成员登录态 + 成员可附加 `isAdmin` + Navbar 用户菜单 | 三个调研项目均无内外有别的权限区分，所有内容对外公开 |
| **DOI/标题自动补全（轻量版）** | 不依赖 AI，纯调 Semantic Scholar + Crossref + DBLP 实现"输入 DOI/标题/标识 → 自动抓元数据 → 确认入库" | LWT 用 Manubot 实现同类能力，我们用多源学术 API + 自建 UI 达到同等效果，且为 Phase 3 AI 版本铺路 |
| **BibTeX 批量导入** | 零依赖自研解析器，支持多类型条目 + LaTeX 转义清理 | al-folio 靠 jekyll-scholar 插件，我们用纯 TS 解析器，前端零依赖 |
| **蜂巢网格成员展示** | `HexGrid` CSS `clip-path` 六边形裁剪 + 蜂巢布局 + 悬停浮层 | 所有调研项目均使用普通矩形卡片网格，六边形布局为原创 |
| **深色科技风** | 全站深色主题 + indigo/cyan 渐变 + 毛玻璃导航 | 三个调研项目均为浅色学术风，深色科技风在学术网站中独树一帜 |
| **智能审核流程（规划）** | Phase 4 的 Moderation Agent 自动预检查并给管理员审核建议 | 超越纯人工静态方案，同时保留人作为最终发布者 |
